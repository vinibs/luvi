Nomenclaturas

O padrão de nomes adotado define que todo e qualquer controller deve possuir a primeira letra de seu nome maiúscula e que deve terminar com o termo "Controller" - sem aspas e também com a letra C maiúscula. Isso se aplica tanto ao arquivo quanto ao nome da classe.

Portanto, para criar um controller que seria originalmente chamado de, por exemplo, "usuario", o nome do arquivo e da classe deve ser "UsuarioController".

Para toda e qualquer classe é recomendado definir seu nome com a primeira letra maiúscula, em sintonia com o padrão dos controllers. Tanto para o arquivo quanto para o nome da classe.

Para views, é recomendado utilizar a primeira letra maiúscula no nome do arquivo.

No banco de dados, cada tabela deve ter exatamente o mesmo nome dos models equivalentes, porém iniciando com letras minúsculas.

Quanto aos atributos das tabelas, o padrão adotado para a chave primária é "id".

Observação: ao realizar uma chamada aos arquivos, como, por exemplo, pelas funções load(), view() ou model(), utilize o nome dos arquivos exatamente como foram definidos (com as letras maiúsculas e minúsculas), para evitar erros ao incluí-los.

Rotas

As rotas sempre encaminharão o visitante para a função "method" da classe "controller", passando os argumentos "id" e "param", quando houver, representados nos exemplos abaixo.


Rotas automáticas

As rotas são definidas automaticamente com base na URL conforme o seguinte padrão: 

http://baseurl/controller/method/param0/param1/param2/...

Portanto, o primeiro parâmetro passado pela URL é referente ao controller, o segundo refere-se ao método e, a partir do terceiro parâmetro, são definidas as variáveis passadas ao sistema por meio do método GET, indexadas numericamente, de forma crescente, a partir do índice zero (0), desde que o método chamado aceite a passagem de parâmetros.


Rotas personalizadas

Podem também ser definidas rotas personalizadas ao editar o arquivo /app/system/Routes.php e realizando chamadas ao método Router:define() conforme o padrão: 

Router::define('URL', 'Controller/Method');

Observe que URL deve ser definida como um caminho relativo a partir da raiz, como nos exemplos abaixo: 

Router::define('user/restrictedarea', 'ExampleController/example');
Router::define('/user/restrictedarea', 'ExampleController/example');
Router::define('/user/restrictedarea/', 'ExampleController/example');
Router::define('user/restrictedarea/', 'ExampleController/example');

Todos os exemplos representam a mesma rota.

Para direcionar uma rota ao método index de um controller, é possível definir o valor da rota como apenas o controller, sem o método. Por exemplo, ao invés de ExampleController/index, é também possível definir como ExampleController e obter o mesmo efeito.

Observação: As rotas mais abaixo no arquivo são ignoradas quando uma rota referente ao mesmo caminho for encontrada, portanto, se houver uma rota '/page/{id}' e, mais abaixo, uma rota '/page/{name}', a primeira será utilizada e a segunda não funcionará.


Rotas com argumentos

É possível determinar quando uma rota deve receber as variáveis GET passadas como argumentos da URL adicionando '{var_name}' (sem aspas) à rota, em que "var_name" representa o nome desejado para a variável, conforme o padrão: 

Router::define('posts/{id}', 'PostsController/show');
Router::define('posts/{id}/edit', 'PostsController/edit');

Dessa forma, o trecho entre chaves ({}) será traduzido para uma variável passada à função determinada na rota.

Os índices no vetor de parâmetros será o nome passado à rota, entre chaves ({}), de forma que, por exemplo, '{id}' gerará um índice "id" no vetor de parâmetros.

Por exemplo, a URL 'posts/12/14/feb/2017', considerando a rota 'posts/{id}/{day}/{month}/{year}', irá resultar em um vetor semelhante a: 

$param = array(
    id => '12'
    day => '14',
    month => 'feb',
    year => '2017',
);

Observe que o nome da variável é definido pelo nome do argumento recebido pelo método do controller que o receberá.

Portanto, se o método for definida como, por exemplo, somemethod ($get) { ... }, o nome do vetor passado será $get, e não $param.


route()

Auxilia na geração de links relativos para dentro do sistema, a partir da raiz do projeto. 

Usage example:
<a href="<?=route('login')?>">...</a>

Assim, o link referenciará o endereço "login" a partir da raiz do projeto. Por exemplo, se o projeto se localizar em "/projetos/luvi", o link apontará para "/projetos/luvi/login".

Por exemplo, quando é encontrado um erro pela função Validation::check() e são retornados, ao formulário, os erros encontrados no preenchimento dos mesmos, são essas mensagens de erro que são exibidas para o usuário.

Para alterar as mensagens de validação de dados, vá até o arquivo app/SystemMessages.php e edite as constantes cujos nomes são iniciads por "msgValidation".

Por exemplo, se, ao analisar um campo nomeado username que é obrigatório e não esteja preenchido, é retornada a seguinte mensagem, sendo que %field% é responsável por exibir o nome do campo:

Controllers

Todos os conrollers criados devem extender o controller principal "Controller". Os controllers fazem a conexão entre os dados armazenados (obtendo-os através dos models) e sua exibição (a partir das views).

Os controllers devem ser armazenados no diretório /app/controllers e devem ter a primeira letra de seu nome maiúscula, seguida da palavra Controller. 

class SomeController extends Controller { ... }

Pode-se, a fim de simplificar a criação de um controller, copiar o arquivo TemplateController.php, do diretório /templates, para o diretório /app/controllers e renomeá-lo (tanto o arquivo quanto a classe) para se adequar às necessidades.

É importante ressaltar que cada função representa o método para o qual a URL direcionará.

Assim, uma função index em um controller qualquer, será responsável por atender a URL quando um método não for definido. Por exemplo, ao entrar com a URL mysite.com/user, a função index do controller User processará a requisição (assumindo-se o uso das rotas automáticas). 

mysite.com/user // Essa URL será processada pela seguinte função:
class UserController extends Controller { 
   public function index () { ... } 
} 

mysite.com/user/register // Essa URL será processada pela seguinte função:
class UserController extends Controller { 
   public function register () { ... } 
}

Página principal

A página principal, raiz do sistema, é exibida a partir da rota padrão para o MainController, utilizando seu método index. Para alterar a página inicial, pode-se alterar a view Default ou o próprio controller, ou mesmo substituí-lo por um novo, com outro nome, e criar uma rota personalizada para a raiz do sistema, apontando para este novo controller. 

mysite.com/ // Essa URL será processada, por padrão, em:
class MainController { 
   public function index () { ... } 
}

getRequest()

Identifica o tipo de requisição realizada para a página (get, post, put, delete, etc.). 

string getRequest ( )

Usage example:
$requestType = $this->getRequest (); // Quando usado de dentro de um controller

Retorna uma string, em letras minúsculas, contendo o tipo de requisição realizada.


Parâmetros GET

Considerando a passagem de parâmetros padrão das rotas com argumentos, as variáveis passadas aos métodos são processadas como trechos da URL, separadas por uma barra. Desta forma, não são utilizadas as variáveis GET com o intuito de manter a URL mais limpa.

Entretanto, para que um método do controller possa fazer uso dos valores vindos da URL, ele deve poder receber argumentos, ou seja, a função deve estar habilitada a receber parâmetros. 

public function something ( $params ) { ... }

Ao utilizar uma variável, por exemplo, $params, para receber os dados vindos da URL em um método, todos os parâmteros passados serão adicionados a ela, de forma que ela passa a ter uma estrutura semelhante à da variável $_GET. Assim, os parâmetros são alcançados utilizando $params["id"], $params["param1"], etc.

Por outro lado, os parâmetros GET ($_GET) continuam podendo ser usados, tendo à disposição apenas a função filterGet() para manuseio destes dados.

Portanto, se utilizar o parâmetro GET tradicional, a URL ficará bagunçada mas os dados serão acessados normalmente. 

URL: mysite.com/controller/method?id=X&something=Y

GET vars:
array(2) {
  ["id"]=>
    string(1) "X"
  ["something"]=>
    string(1) "Y"
}

Inclusão de componentes

load()

Carrega um arquivo, passado o caminho a partir da raiz da aplicação. Caso não consiga encontrar tal arquivo, gera um Fatal Error. 

bool load ( string $pathToFile )

Usage example:
load ('/app/lib/anything.php');

Retorna TRUE se o arquivo já tiver sido carregado ou FALSE caso contrário.


view()

Carrega uma view, dado o nome do arquivo sem a extensão. Caso não consiga encontrar tal arquivo no diretório /app/views, gera um Fatal Error.

Pode receber uma variável de dados para usar no arquivo de view, passado e acessado sempre por $data, conforme citado em passagem de dados. 

void view ( string $viewName, [$data = NULL] )

Usage example:
view ('LoginForm');
view ('somePage', $viewData);

Não possui retorno.


model()

Carrega um model, dado o nome do arquivo sem a extensão. Caso não consiga encontrar tal arquivo /app/models, gera um Fatal Error.

Por padrão todos os models são carregados automaticamente, portanto, essa função não se faz muito necessária. 

bool model ( string $viewName )

Usage example:
model ('Animal');

Retorna TRUE se o arquivo for carregado corretamente ou FALSE caso contrário.


asset()

Gera o caminho para, dado o nome do arquivo com a extensão, a partir do diretório /assets/.

Útil para a inclusão de arquivos como CSS e JS nas views. 

void asset ( string $assetFile )

Usage example:
<link href="<?=asset ('style/style.css')?>">

Retorna uma string contendo o caminho para o arquivo a partir da raiz do projeto.



Views

As views são responsáveis pela exibição dos dados.

Não há um padrão definido para a forma como as views são utilizadas. Elas apenas devem ser armazenadas no diretório /app/views.


Inclusão de views

Quando for necessário usar uma view no sistema, deve-se utilizar a função view() para carregar o arquivo que contém as informações necessárias.

Ao incluir um arquivo através desta função, será automaticamente procurado e incluído, a partir do diretório /app/views, um arquivo que possua o nome indicado e tenha a extensão .php, .html ou .phtml.

Também é possível passar dados para uma view, adicionando o segundo parâmetro à sua chamada. O dado passado fica acessível na view pelo nome $data. 

public function someFunction () {
...
return view('some');
return view('another', $someData);
}

Observação: utilizar o return ao carregar uma view é opcional, mas pode ser útil para garantir que não haverá código sendo executado após o carregamento da tela do usuário.


Passagem de dados

É possível passar uma variável para a view através da função view(), passando a variável desejada como segundo argumento.

Caso haja a necessidade de passar mais de uma variável para a view, é preciso colocar as múltiplas variáveis a serem passadas em um array e então passar esse array para a função view().

Note que toda view, ao receber uma variável, tem seu nome definido como $data. Assim, para acessar qualquer valor enviado para a view, deve-se usar esta variável. 

Passando um valor ou variável única:
view('someView', 'Fruits');
Na view, para se obter o valor, usa-se:
$data

Passando um array:
view('someView', [$var1, $var2, var3, ... ]);
Na view, para se obter os valores, usa-se:
$data[0] // Contém o valor de $var1
$data[1] // Contém o valor de $var2
...

Templates nas views

Também é possível elaborar templates. Para isso, pode-se criar um arquivo com o HTML base da página e, onde o conteúdo for variável, trocar o texto por !!section('sectionname]')!!, em que sectionname representa o nome que identifica a seção (por exemplo, uma seção identificada por "body" seria especificada como !!section('body')!!.

Observação: quando fazendo uso do sistema de tamplates, ao incluir uma view, deve-se usar o método view() para inclui-la, passando como parâmetro diretamente a view que se deseja carregar, e não o template, caso seja usado a funcionalidade de extender o template na view em questão.


Template

Para criar um template, pode-se criar um arquivo com o HTML base da página e, onde o conteúdo for variável, trocar o texto por !!section('sec_name')!!, onde sec_name representa o nome identificador da seção, como no exemplo, em que é definida uma seção chamada de "header": 

<html>
   <head>
       ...
   </head>
   <body>
       ...
       !!section('header')!!
   </body>
</html>
Extender template

Quando dentro do contexto de templates, arquivos que contenham os conteúdos variáveis devem extender um arquivo de template (conforme a estrutura dada em template.

Para fazer isso, no arquivo HTML que contém os dados variáveis, deve-se incluir o trecho @extends('template'), em que template representa o nome do arquivo (sem a extensão), preferencialmente no início do arquivo, por uma questão de organização. 

@extends('template') 
...
[Resto dos códigos da página]

Ao referenciar o template, o sistema buscará o arquivo a partir do diretório em que a view passada para o método se encontra. Por exemplo, se a view estiver em /app/views e o templete estiver em outro diretório, como em /app/views/templates, deve-se definir o caminho para o template, na definição do @extends.

Abaixo um exemplo desta situação, considerando um exemplo em que o arquivo de template se chama main.php: 

@extends('templates/main') 
...
[Resto dos códigos da página]
Seções de dados

Representam os blocos de código HTML que substituirão os elementos !!section('...')!! do template que possuam o mesmo nome..

São definidos na view por um início dado por @section('sectionname'), onde sectionname representa o nome da seção - que deve ser igual ao definido no elemento !!section('...')!! no template base -, e finalizados por @endsection. 

@extends('templates/main') 

                
@section('div1')

                
   <div class="..."></div>

                
@endsection

            
            

                Cada bloco de seção será substituído pelo !!section('...')!! no template que possua o mesmo
                nome identificador definido no bloco @section('...').
            

            

                O exemplo acima funcionará apenas para uma seção chamada 'div1'. Por exemplo, tomemos o
                arquivo templates/main.php como composto pelo seguinte código:
            

            
                
<html>

                
   <head>

                
       <title>Main Template</title>

                
   </head>

                
   <body>

                
       !!section('div1')!!

                
   </body>

                
</html>

            
            

                Assim, é preciso que o nome indicador ('div1', no exemplo acima) seja igual tanto na seção
                definida no template quanto no bloco que define a seção na view que extende o template.

Chave primária

		

			Por padrão, é considerado que a coluna de uma tabela no banco de dados usada como chave
			primária (primary key) chama-se "id".
		

		

			Caso o nome da chave primária de uma tabela seja diferente, pode-se definir, no próprio
			model, a variável public $primarykey, a qual indica o nome do campo a ser usado
			para buscas por valores únicos na tabela de nome igual ao da classe.
		

		
			
class Animal extends Model { 

			
    public $primarykey = 'animal_id'; 

			
}

		
		

			Desta forma, ao relizar uma busca pela função get(), a consulta 
			procurará pelo valor passado na coluna denominada "animal_id", ao invés de procurar por uma
			coluna chamada "id" e buscar tal valor nela.

Consultas

		

			get()
			

				Procura na tabela definida pelo objeto por um dado cuja chave primária seja igual a 
				$id.
			

			
				
object get ( $id )

				

Usage example:

				
Fruit::make()->get (5); // Procura na tabela Fruit pela primeira linha cujo valor da chave primária seja igual a 5

				
Product::make()->get ('rice'); // Procura na tabela Product pela primeira linha cujo valor da chave primária seja igual a 'rice

			
			

				Retorna um objeto do tipo da classe que originou a chamada à função.
			

			

			all()
			

				Seleciona todos os dados da tabela definida pelo objeto.
			

			
				
object_array all ( )

				

Usage example:

				
Juice::make()->all (); // Retorna um vetor de objetos do tipo Juice com todos os dados armazenados na tabela de mesmo nome

			
			

				Retorna um vetor de objetos do tipo da classe que originou a chamada à função.
			

			

			select()
			

				Define o que deve ser buscado no banco de dados.
			

			

				Quando executado sozinho junto à função find(),
				procura pelos dados recebidos por parâmetro (ou por todos os dados, quando o parâmetro
				não for recebido ou definido) sem uma condição de restrição (where) na tabela do
				objeto pelo qual a função é chamada.
			

			
				
object:DB select ( [string $selectedData = NULL] )

				

Usage example:

				
Person::make()->select ('id, name, lastname')->find (); // Seleciona apenas 'id', 'name' e 'lastname' da tabela Person

				
Car::make()->select ('*')->find (); // Seleciona todos os dados da tabela Car

				
Animal::make()->select ()->find (); // Seleciona todos os dados da tabela Animal

			
			

				Retorna um objeto da classe DB.
			

			

				Observação: esta função deve ser executada SEMPRE seguida da função find(). 
				Caso contrário, a consulta não será executada.
			

			

			join()
			

				Define uma união com outra tabela do banco de dados.
			

			

				Realiza a união com a tabela $joinTable, comparando a coluna $joinTableCol,
				da tabela a ser unida, com a coluna $thisTableCol, tabela ao qual o objeto atual
				pertence, utilizando a comparação $compare (por padrão, verifica se são iguais).
			

			

				Ao executar a consulta, utilizando o método 
				find(), os resultados
				obtidos são apenas referentes ao  objeto que originou a chamada da
				função, e, portanto, os dados das colunas "incluídas" não são
				trazidos com o objeto.
			

			

				Em outras palavras, ao chamar a função a partir de um objeto, por exemplo, Device e uni-lo
				com um objeto Person, o resultado será um objeto da classe Device, com seus atributos
				tradicionais (manipulados pelos métodos getters e setters) mais os atributos da classe
				Person, públicos (manipuláveis diretamente, sem necessitar dos métodos get e set).
			

			
				
object:DB join ( string $joinTable, $joinTableCol, $thisTableCol, [$compare] )

				

Usage example:

				
Television::make()->join ('hometheater', 'hometheater.id_tv', 'television.id')->where ('hometheater.video = ?', 'hdmi')->find (); // Procura por toda a tabela Television, unindo-se com a tabela Hometheater, por uma TV cujo HT tenha a propriedade 'video' igual a 'hdmi' 

			
			

				Retorna um objeto da classe DB.
			

			

				Observação: esta função deve ser executada SEMPRE antes da função where()
				(quando aplicável). Caso contrário, a consulta não será executada 
				corretamente.
			

			

			where()
			

				Define a condição de restrição para a consulta dada pela função 
				select() ou inicia uma nova busca por todas as colunas da tabela
				seguindo a condição de restrição passada.
			

			

				A condição de busca ($where) é composta de uma string com o código SQL do where
				(exceto pela palavra chave "WHERE"), com os valores a serem buscados trocados por 
				uma "?", enquanto que $whereVars, quando passada, pode ser uma string contendo 
				o único valor a substituir a "?" na string $where ou um vetor com todos os elementos
				que substituirão, respectivamente, cada uma das interrogações do código passado.
			

			
				
object:DB where ( string $where, [$whereVars] )

				

Usage example:

				
Cat::make()->where ('name = ?', 'Felix')->find (); // Procura por toda a tabela Cat por um valor cujo atributo 'name' seja 'Felix'

				
Dog::make()->select ('id, age')->where ('name = ? AND weight > ?', [$name, $weight])->find (); // Seleciona 'id' e 'age' na tabela Dog, procurando por uma linha em que 'name' seja igual ao valor de $name e 'weight' seja maior que o valor de $weight

				
Lion::make()->select ('*')->where ('wild = TRUE')->find (); // Seleciona tudo na tabela Lion, procurando por uma linha em que 'wild' seja TRUE

			
			

				Retorna um objeto da classe DB.
			

			

				Observação: esta função deve ser executada SEMPRE depois da função select()
				(quando aplicável) e seguida da função find(). Caso contrário, a consulta não
				será executada.
			

			

			orderBy()
			

				Define o campo pelo qual os resultados da consulta serão ordenados em $orderValue
				e a forma de ordenação dos dados (ASC ou DESC) em $order.
			

			

				Pode ser usada várias vezes em uma mesma consulta para definir os vários campos pelos
				quais a consulta deverá ordenar os dados.
			

			
				
object:DB orderBy ( string $orderValue, [string $order = 'ASC'] )

				

Usage example:

				
People::make()->select ()->orderBy ('name', 'DESC')->find (); // Ordena o resultado da busca pelo campo 'name' de forma descendente (DESC)

				
Dog::make()->select ('id, name')->where ('weight > ?', $weight)->orderBy ('name')->orderBy ('weight', 'DESC')->find (); // Ordena os resultados pelo atributo 'name' de forma ascendente e, posteriormente, pelo atributo 'weight' de forma descendente (DESC)

			
			

				Retorna um objeto da classe DB.
			

			

				Observação: esta função deve ser executada SEMPRE após as funções select() e where(),
				em qualquer situação, e seguida da função find(). Caso contrário, a consulta não será 
				executada.
			

			

			extraSql()
			

				Adiciona código SQL à consulta.
			

			

				Caso seja necessário adicionar código SQL à consulta, como um JOIN ou outro comando,
				a função realiza este procedimento antes da execução da consulta.
			

			
				
object:DB extraSql ( string $sql )

				

Usage example:

				
Television::make()->select ()->extraSql ('Some SQL')->find (); // Acrescenta o SQL à consulta

				
Printer::make()->select ('id, code')->where ('code < ?', $code)->orderBy ('name')->extraSql ('Another SQL')->find (); 

			
			

				Retorna um objeto da classe DB.
			

			

				Observação: esta função deve ser executada SEMPRE após as funções select() ou where(),
				em qualquer situação, e seguida da função find(). Caso contrário, a consulta não será 
				executada.

object save ( )

Classe DB

		

			Também é possível fazer consultas, inserções e exclusões no banco de dados a partir de
			funções da classe DB, ao invés de fazê-las a partir de objetos. 
		

		

			Essa opção pode ser útil caso seja necessário buscar em uma tabela diferente da do objeto
			que esteja em uso no momento.
		

		

			As funções desta classe são exatamente as mesma descritas acima, exceto pela função
			get().
		

		

			Os parâmetros de algumas das funções são diferentes para a classe DB. As mudanças são
			descritas a seguir.
		

		

			DB->select()
			

				As mesmas propriedades da função select() descrita
				anteriormente, com a diferença de que esta exige o nome da tabela como parâmetro 
				adicional.
			

			
				
object:DB select ( [string $selectedData = NULL], string $table )

				

Usage example:

				
DB::make()->select ('id, name, lastname', 'Person')->find ();

				
DB::make()->select ('*', 'Car')->find ();

				
DB::make()->select (NULL, 'Animal')->find ();

			
			

				Retorna um objeto do tipo DB.
			

			

			DB->join()
			

				As mesmas propriedades da função join() descrita
				anteriormente, com a diferença de que deve ser executada SEMPRE após um 
				select().
			

			
				
object:DB join ( string $joinTable, $joinTableCol, $thisTableCol, [$compare] )

				

Usage example:

				
DB::make()->select ('name')->join ('hometheater', 'hometheater.id_tv', 'television.id')->where ('hometheater.video = ?', 'hdmi')->find (); // Procura por toda a tabela Television, unindo-se com a tabela Hometheater, por uma TV cujo HT tenha a propriedade 'video' igual a 'hdmi' 

			
			

				Retorna um objeto do tipo DB.
			

			

			DB->where()
			

				As mesmas propriedades da função where() descrita
				anteriormente, com a diferença de que esta exige o nome da tabela como parâmetro 
				adicional, quando não usada em conjunto com a função select().
			

			
				
object:DB where ( string $where, [$whereVars], [string $table] )

				

Usage example:

				
DB::make()->where ('name = ?', 'Felix', 'Cat')->find ();

				
DB::make()->select ('id, age', 'Dog')->where ('name = ? AND weight > ?', [$name, $weight])->find (); // Seleciona 'id' e 'age' na tabela Dog, procurando por uma linha em que 'name' seja igual ao valor de $name e 'weight' seja maior que o valor de $weight

				
DB::make()->select ('*', 'Lion')->where ('wild = TRUE')->find (); // Seleciona tudo na tabela Lion, procurando por uma linha em que 'wild' seja TRUE

			
			

				Retorna um objeto do tipo DB.
			

			

			find()
			

				Executa a operação iniciada pelas funções select() e 
				where(). Deve ser usada apenas depois de pelo menos
				alguma das funções citadas. 
			

			
				
object:DB find ( )

				

Usage example:

				
DB::make()->select ('id, age', 'Cat')->find (); // Executa a consulta select anterior

				
DB::make()->select ('id, age', 'Cat')->where ('name = ? AND weight > ?', [$name, $weight])->find (); // Executa a consulta select/where anterior

				
DB::make()->where ('name = ? AND weight > ?', [$name, $weight])->find (); // Executa a consulta select/where anterior

			
			

				Retorna o objeto salvo em caso de sucesso ou NULL caso contrário.
			

			

			generatedSql()
			

				Retorna o código SQL gerado pelas demais funções da classe DB.
			

			
				
object:DB generatedSql ( )

				

Usage example:

				
DB::make()->select ('id, age', 'Cat')->generatedSql (); // Retorna o SQL gerado para esta consulta

			
			

				Retorna uma string com o código SQL gerado pelas demais funções da classe.
			

			

			DB::all()
			

				Recebe uma string $table por parâmetro e busca no banco por todos os dados da tabela passada.
			

			
				
object all ( $table )

				

Usage example:

				
DB::all ($mouse); // Busca por todos os dados de uma dada tabela

			
			

				Retorna os objetos encontrados em caso de sucesso ou NULL caso contrário.
			

			
DB::save()
			

				Recebe um objeto $object por parâmetro e insere ou atualiza seus dados no banco.
			

			
				
object save ( $object )

				

Usage example:

				
DB::save ($objAnimal); // Insere ou atualiza os dados atuais do objeto $objAnimal do banco

			
			

				Retorna o objeto salvo em caso de sucesso ou a mensagem de erro obtida caso contrário.
			

			

			DB::delete()
			

				Recebe um objeto $object por parâmetro e exclui seus dados do banco.
			

			
				
bool delete ( $object )

				

Usage example:

				
DB::delete ($objFruit); // Remove os dados atuais do objeto $objFruit do banco

			
			

				Retorna TRUE em caso de sucesso ou FALSE caso contrário.
			

		



		


		

		
Autenticação

		

			Para facilitar o processo de autenticação e simplificar o sistema de login
			e registro de usuários, há a classe Auth.
		

		

			Todas as funções dessa classe são estáticas.
		

		

			Observação: ao utilizar esta classe, deve-se lembrar que o campo destinado à senha do
			usuário é criptografado dentro das funções nativas da classe. Para que seus métodos
			funcionem corretamente, ao criar um model para o objeto usuário
			é preciso definir que na função setPassword() (ou equivalente destinada a preencher o
			campo da senha) a atribuição ao valor será feita através da função 
			hashPassword().
		

		
			
public function setPassword ($password) {

			
    $this->password = Auth::hashPassword($password);

			
    return $this;

			
}

		
		

		
Auth::doLogin()

		

			Recebe os dados do usuário $authData e realiza o login, retornando à página 
			anterior e enviando os erros encontrados e os valores dos campos (vide 
			withErrors() e withValues()) para
			que sejam exibidos ao usuário, ou realizando a criação da sessão (vide 
			createAuthSession()) para o usuário, redirecionando-o
			para uma página especificada por $redirectLocation (ou para a página inicial, caso
			não seja passado o valor) e buscando os dados na tabela	$dbTable (definida como
			'User' caso não seja passada) para fazer a validação.
		

		

			Observação: essa função utiliza o índice do vetor passado como parâmetro para identificar
			o atributo na tabela do banco de dados a ser comparado. Por isso, é essencial que o 
			formulário tenha em seus campos o atributo name igual ao nome das suas respectivas
			colunas no banco, conforme citado em Banco de Dados.
		


		
			
bool doLogin ( array $authData, [string $redirectLocation = NULL], [string $dbTable = 'User'] )

			

			
Usage example:

			
Auth::doLogin (filterPost(), 'logged/home', 'Users');

			
Auth::doLogin ($data); // Ao final redireciona para a raiz (/) e procura os dados na tabela 'User'

			
		

			Retorna TRUE caso o login seja bem sucedido ou FALSE caso contrário.

Redefinição de senha

		

			Há também um conjunto de funções criadas para facilitar o processo de redefinição de senha
			do usuário. Estas funções exigem que o banco de dados possua algumas características.
		

		

			Para que funcionem as funções, é preciso que o banco de dados tenha uma tabela específica
			para armazenar as requisições de redefinição de senha. Essa tabela deve conter, no
			mínimo, três colunas, sendo elas email (referente ao e-mail do usuário, que deve
			estar disponível na tabela de usuários), token (o código alfanumérico de
			redefinição gerado automaticamente) e val (indica se o código já foi usado ou não).
			É um sistema bastante simples, que não envolve tempo para expiração de código ou métodos 
			mais sofisticados.
		

		
			
CREATE TABLE passwordreset (

			
    id INT ... ,

			
    email VARCHAR ... ,

			
    token VARCHAR ... ,

			
    val BOOL ... 

			
);

			
		

			Esta tabela é o suficiente para que o mecanismo de recuperação de senha funcione.
		

		

			Observação: o nome das três colunas citadas DEVE ser exatamente como descrito. Caso
			contrário, a redefinição de senha não funcionará.
		

		

		

			Auth::requestPassReset()
